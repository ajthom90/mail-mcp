# v0.3 — Realtime events / push notifications Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Surface mail events (new message, thread updated) from the daemon to the MCP host in realtime, so models can `wait_for_reply` / `wait_for_match` and proactively react.

**Architecture:** Per-provider watchers (Gmail history poll first, Graph delta poll first; push subscriptions deferred to v0.3.1) feed a shared `mail-mcp-core::events::EventBus`. The bus fans out to (a) the existing IPC notification channel for tray subscribers and (b) two new MCP primitives — `mail://account/{id}/inbox/new` resource subscriptions and the `wait_for_*` blocking tools.

**Tech Stack:** Existing workspace deps. New: `tokio::sync::broadcast` for the bus (already pulled in transitively), `cron` for the inevitable scheduled coalescing window, `wiremock` for tests. No new external crates beyond what's already there.

```rust
use tokio::sync::broadcast;
use crate::events::Event;

#[derive(Clone)]
pub struct EventBus {
    tx: broadcast::Sender<Event>,
}

impl EventBus {
    pub fn new(capacity: usize) -> Self {
        let (tx, _) = broadcast::channel(capacity);
        Self { tx }
    }

    pub fn publish(&self, event: Event) {
        // Lagged subscribers will see RecvError::Lagged on their next recv;
        // we don't block the publisher waiting for slow consumers.
        let _ = self.tx.send(event);
    }

    pub fn subscribe(&self) -> broadcast::Receiver<Event> {
        self.tx.subscribe()
    }
}
```

```rust
#[tokio::test]
async fn debounces_same_thread_within_5s() {
    let bus = EventBus::new(64);
    let coalesce = Coalescer::new(bus.clone(), Duration::from_secs(5), 10);
    let mut sub = bus.subscribe();

    let evt = || Event::MailThreadUpdated {
        account_id: "a".into(), thread_id: "t1".into(), message_id: "m".into()
    };
    coalesce.publish(evt());
    coalesce.publish(evt());
    coalesce.publish(evt());

    // Only the first should escape the debounce window.
    let first = tokio::time::timeout(Duration::from_millis(100), sub.recv()).await.unwrap().unwrap();
    assert!(matches!(first, Event::MailThreadUpdated { .. }));

    let no_more = tokio::time::timeout(Duration::from_millis(100), sub.recv()).await;
    assert!(no_more.is_err(), "second event should be debounced");
}
```

## Phase C — Provider push subscriptions (deferred to v0.3.1)

Tasks 8-12 implement Gmail Pub/Sub push and Microsoft Graph webhook subscriptions, replacing the polling watchers when the user has the necessary infrastructure (GCP project for Gmail, public webhook URL for Graph). The poll-mode watchers stay as the always-available fallback.

These are deferred to v0.3.1 because:
- Gmail Pub/Sub requires the user to set up a GCP project + Pub/Sub topic, not zero-config.
- Graph webhooks need a public URL the daemon doesn't have without a tunnel.
