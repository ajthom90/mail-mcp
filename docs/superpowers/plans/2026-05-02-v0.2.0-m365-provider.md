# v0.2.0 — M365 Provider Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add Microsoft 365 / Outlook as a second provider behind the existing `MailProvider` trait, using Microsoft Graph API and OAuth 2.0 PKCE.

**Architecture:** New `crates/mail-mcp-core/src/providers/m365/` module mirroring the gmail/ structure. New `oauth::microsoft` module with the Graph PKCE config. Daemon's `accounts.add_oauth` IPC method grows a `"m365"` provider value; everything downstream (permissions, dispatch, MCP tools) is provider-agnostic and needs no changes.

**Tech Stack:** Existing workspace deps. `reqwest` for HTTP, `serde` for the wire types, `wiremock` for tests. No new external crates.

- [ ] **Step 1: oauth/microsoft.rs**

```rust
//! Microsoft Graph OAuth 2.0 endpoints + Mail scope set.

use super::ProviderConfig;

/// Build a ProviderConfig for Microsoft Graph with the given OAuth client_id.
///
/// `/common` accepts both consumer (outlook.com, hotmail.com, live.com) and
/// AAD work/school accounts. The user picks during the consent prompt.
///
/// `offline_access` is required for the refresh_token. Microsoft rotates
/// refresh tokens on every refresh — the AuthClient's RefreshRotationCallback
/// (issue #2 fix) handles persistence transparently.
pub fn config(client_id: impl Into<String>) -> ProviderConfig {
    ProviderConfig {
        auth_url: "https://login.microsoftonline.com/common/oauth2/v2.0/authorize".into(),
        token_url: "https://login.microsoftonline.com/common/oauth2/v2.0/token".into(),
        client_id: client_id.into(),
        default_scopes: vec![
            "offline_access".into(),
            "https://graph.microsoft.com/Mail.ReadWrite".into(),
            "https://graph.microsoft.com/Mail.Send".into(),
            "https://graph.microsoft.com/User.Read".into(),
        ],
    }
}

#[cfg(test)]
mod tests {
    use super::*;

    #[test]
    fn microsoft_config_has_expected_endpoints() {
        let cfg = config("test-client-id");
        assert_eq!(
            cfg.auth_url,
            "https://login.microsoftonline.com/common/oauth2/v2.0/authorize"
        );
        assert_eq!(
            cfg.token_url,
            "https://login.microsoftonline.com/common/oauth2/v2.0/token"
        );
        assert!(cfg.default_scopes.iter().any(|s| s == "offline_access"));
        assert!(cfg.default_scopes.iter().any(|s| s.contains("Mail.ReadWrite")));
        assert!(cfg.default_scopes.iter().any(|s| s.contains("Mail.Send")));
    }
}
```

### Task 3: parse.rs — Graph message JSON → domain Message

```json
{
  "id": "AAMkAD...",
  "subject": "Hello",
  "bodyPreview": "Hi there",
  "from": { "emailAddress": { "name": "Alice", "address": "alice@example.com" } },

  "ccRecipients": [...],
  "categories": ["Followups"],
  "isRead": false,

  "parentFolderId": "AAAA...",

  "body": { "contentType": "html" | "text", "content": "..." },

}
```

### Task 4: folders.rs — list_folders + list_labels

`list_folders`: `GET /me/mailFolders?$top=100`. Filter for `wellKnownName` matching `inbox|sentitems|deleteditems|drafts|junkemail|archive`. Map to our `Folder` with the well-known name as both id and name.

`list_labels`: `GET /me/outlook/masterCategories`. Return user-defined categories.

### Task 5: messages.rs — search, get_thread, get_message

`search_impl`: `GET /me/messages?$search="<text>"&$top=<limit>` with `$skiptoken=<cursor>` from a previous response's `@odata.nextLink`. Returns `SearchResults` with `next_cursor` set to the full nextLink URL when present.

`get_thread_full`: `GET /me/conversations/{conversationId}/messages?$expand=microsoft.graph.message`. Returns a `Thread` with all messages decoded.

`get_message_impl`: `GET /me/messages/{id}`.

### Task 6: triage.rs — mark_read / star / label / move_to / archive

All map to `PATCH /me/messages/{id}` with appropriate body or `POST /me/messages/{id}/move`. See spec for the exact endpoint table.

### Task 7: compose.rs — drafts + send

- `create_draft`: `POST /me/messages` with `isDraft: true` and the message body in Graph's JSON shape (NOT raw MIME — Graph builds the MIME for us).
- `update_draft`: `PATCH /me/messages/{id}`.
- `send_message`: `POST /me/sendMail` — creates and sends in one call.
- `send_draft`: `POST /me/messages/{id}/send`.

## Phase C — Wiring

### Task 14: Daemon recognizes provider="m365"

**Files:**
- Modify: `crates/mail-mcp-daemon/src/ipc_handler.rs`

In `add_oauth`:

```rust
let cfg = match provider.as_str() {
    "gmail" => self.oauth_cfg_google.clone(),
    "m365"  => self.oauth_cfg_microsoft.clone(),
    other => return Err(CoreError::Provider(format!(
        "provider not supported: {other}"
    ))),
};
let challenge = oauth::begin_authorization(&cfg, None).await?;
```

`DaemonHandler` grows a second oauth config field. Daemon entrypoint accepts `--microsoft-client-id` alongside `--google-client-id`.

In `complete_oauth` and `hydrate_providers`, branch on `account.provider` to construct either a `GmailProvider` or an `M365Provider`. The fetched user-email comes from a different endpoint per provider:
- Google: `GET https://openidconnect.googleapis.com/v1/userinfo` → `email`
- Microsoft: `GET https://graph.microsoft.com/v1.0/me` → `mail` or `userPrincipalName`
