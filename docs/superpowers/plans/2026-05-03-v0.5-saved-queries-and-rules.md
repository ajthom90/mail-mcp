# v0.5 — Saved queries + automation rules Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Two related capabilities: (1) named saved queries usable from MCP tools and as resource paths, (2) user-installable automation rules ("when X arrives, do Y") that fire under the existing per-account permission gate.

**Architecture:** Saved queries live in `${XDG_CONFIG_HOME}/mail-mcp/saved-queries.toml`; rules in `rules.toml`. The daemon loads both at startup, exposes them via IPC for the tray to manage, and runs a per-rule worker that subscribes to the v0.3 EventBus. Rule actions go through the same permission gate as direct tool calls — rules NEVER bypass approvals.

**Tech Stack:** Existing workspace deps. New: `cron` for scheduled triggers (`schedule.hourly`, `schedule.daily`), `notify` for hot-reloading the TOML files when the user edits them. `wiremock` for tests.

## Phase B — Automation rules

### Task 5: Rule type + storage

**Files:**
- Create: `crates/mail-mcp-core/src/rules/mod.rs`
- Create: `crates/mail-mcp-core/src/rules/store.rs`

```rust
#[derive(Debug, Clone, Serialize, Deserialize, PartialEq)]
pub struct Rule {
    pub name: String,
    pub account: String,                // ID or "*"
    pub trigger: Trigger,
    #[serde(rename = "match")]
    pub matcher: Option<String>,        // query expression; required for mail.new + calendar.new_invite
    pub schedule_time: Option<String>,  // "HH:MM" for schedule.daily; required for that trigger
    pub actions: Vec<RuleAction>,
    pub enabled: bool,
}

#[derive(Debug, Clone, Copy, Serialize, Deserialize, PartialEq)]
#[serde(rename_all = "snake_case")]
pub enum Trigger {
    MailNew,
    CalendarNewInvite,
    ScheduleHourly,
    ScheduleDaily,
}

#[derive(Debug, Clone, Serialize, Deserialize, PartialEq)]
pub struct RuleAction {
    pub tool: String,
    pub args: serde_json::Value,
}
```

- [ ] **Step 1: Validation rules**
  - `matcher` required for `MailNew` / `CalendarNewInvite`; rejected for schedule triggers.
  - `schedule_time` required for `ScheduleDaily`; rejected for others (hourly fires every hour at :00).
  - `actions` non-empty.
  - `tool` is in the action allow-list (see Task 6).

- [ ] **Step 2: TOML round-trip tests**

### Task 7: Rule worker

**Files:**
- Create: `crates/mail-mcp-core/src/rules/worker.rs`

One `tokio::task` per enabled rule. Subscribes to:
- `MailNew` / `CalendarNewInvite` triggers — EventBus subscription with matcher filter.
- `ScheduleHourly` — `tokio::time::interval(Duration::from_secs(3600))`.
- `ScheduleDaily` — calculates next run from `schedule_time` in the user's local TZ, sleeps until then, fires, recomputes.

For each fire:
1. Build the action context (event payload for triggered rules; empty for scheduled).
2. Dispatch each action sequentially via the existing tool-dispatch pipeline.
3. If a tool is permission-gated, the daemon parks the action and emits `approval.requested` with `triggered_by_rule: "<name>"` set; the user approves through the tray.
4. Log the firing to `${XDG_DATA_HOME}/mail-mcp/rules.log`.

- [ ] **Step 1: Worker harness + shutdown signal**
- [ ] **Step 2: MailNew + CalendarNewInvite trigger handling + tests**
- [ ] **Step 3: Schedule-trigger handling with clock mocking + tests**
- [ ] **Step 4: Approval-gated action plumbing — verify `triggered_by_rule` field flows through to the tray**
- [ ] **Step 5: Commit**

```bash

```
