# v0.1d — Linux Tray App Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Ship a Linux tray app for the mail-mcp daemon, feature-equivalent to the v0.1b macOS / v0.1c Windows trays.

**Architecture:** New `tray-app-linux/` directory built with Meson + GCC. C application using GTK4 + libadwaita + AyatanaAppIndicator3. The daemon binary is bundled inside the install prefix (`<prefix>/bin/mail-mcp-daemon`). Tray holds zero persistent state — every fact comes from the daemon over UDS JSON-RPC (the existing v0.1a Unix IPC code path; no changes needed).

**Tech Stack:** C11, GTK4, libadwaita, AyatanaAppIndicator3, json-glib, Meson, CMocka (tests). CI runs on `ubuntu-latest`. Linux daemon ELF produced via `cargo build -p mail-mcp-daemon`. Distribution via tarball + Flatpak; per-distro packaging is left to downstream.

```bash
#!/usr/bin/env bash
# Build the mail-mcp-daemon Rust binary and copy it to the meson custom_target
# output path. Honors host architecture; cross-compile is handled by the CI
# runner setting CARGO_TARGET=*.
set -euo pipefail
out="$1"

cd "$(dirname "$0")/../.."

target_flag=()
if [[ -n "${CARGO_TARGET:-}" ]]; then
  rustup target add "$CARGO_TARGET" >/dev/null
  target_flag=(--target "$CARGO_TARGET")
  built="target/$CARGO_TARGET/release/mail-mcp-daemon"
else
  built="target/release/mail-mcp-daemon"
fi

cargo build --release -p mail-mcp-daemon "${target_flag[@]}"
install -m 0755 "$built" "$out"
```

### Task 6: ipc/ipc_client.c — UDS JSON-RPC client

**Files:**
- Create: `tray-app-linux/src/ipc/ipc_client.h`
- Create: `tray-app-linux/src/ipc/ipc_client.c`
- Create: `tray-app-linux/tests/test_ipc_client.c`
- Modify: `tray-app-linux/tests/meson.build`

UDS client integrated with `GMainContext`. Three operations:
1. `mailmcp_ipc_call_async()` — fire-and-forget RPC; the callback runs on the GTK main loop with the response `JsonNode*`.
2. `mailmcp_ipc_subscribe()` — sends `subscribe(events: [...])` and AWAITS the ack before registering the local handler. (Same fix as v0.1b/v0.1c: race window if you register first.)
3. `mailmcp_ipc_disconnect()` — close the socket cleanly.

Internally, an `IO`-direction `GSource` owned by the client reads from the socket FD, accumulates bytes, scans for `\n`, parses each frame as JSON-RPC. Outstanding requests are tracked in a `GHashTable` keyed by `id`. Notifications are dispatched by method name to the per-event-type list of subscribers.

- [ ] **Step 2: Implementation outline**

The transport uses a `GSocket` (`G_SOCKET_FAMILY_UNIX`, `G_SOCKET_TYPE_STREAM`) connected to `socket_path`. After connect, attach `g_socket_create_source()` with condition `G_IO_IN` to the main context; the read callback reads into a heap-allocated buffer and emits one JSON-RPC frame per `\n`.

For each parsed frame:
- If it has `id`, look up in `pending_requests` table → call response callback → remove entry.
- If no `id`, lookup `method` in `subscriber_table` → call each subscriber.

`mailmcp_ipc_subscribe` uses `mailmcp_ipc_call_async` to send the subscribe RPC, registers the callback in the subscriber table inside the response callback (after ack), and returns immediately. (Synchronous-feeling subscribe is implemented via `g_main_context_iteration()` until the response callback fires — see implementation.)

- [ ] **Step 3: Mock UDS server for tests**

`tray-app-linux/tests/mock_uds_server.c` (new helper, included via tests/meson.build): listens on a temp `socket()`, accepts one connection, runs a child thread that echoes scripted responses to scripted requests. Pattern is the same as the C# `MockIpcServer` from v0.1c, with stdin/stdout swapped for socket I/O.

- [ ] **Step 4: Write three tests**

- `test_status_call` — stub the server to respond to `{"method":"status",...}` with the canonical status payload; assert the parsed `MailMcpStatus` matches.
- `test_rpc_error_propagates` — stub the server to respond with `{"error":{"code":-32000,"message":"boom"}}`; assert the response callback receives a non-NULL `GError` with that message.
- `test_subscribe_receives_notification` — stub the server to ack the subscribe RPC then push a `account.added` notification; assert the per-event callback fires with the account payload.
