# mail-mcp v0.4 — Calendar integration design

## Goals

Add calendar reading + writing to the existing per-account provider infrastructure. Email and calendar are tightly coupled in real workflows ("schedule a follow-up after this thread", "what's on my calendar tomorrow when I read this invite"), and v0.4 lets a model use both inside one MCP session.

Provider parity from day one: Gmail accounts get Google Calendar, M365 accounts get Outlook Calendar. IMAP-only accounts (added in v0.2) have no calendar — that's expected.

## Non-goals

- A separate calendar-only mode. Calendar capability is part of the existing account; if you connect a Gmail account you get its calendar(s).
- CalDAV / iCalendar generic-protocol support (separate provider, deferred).
- Exchange on-prem / EWS support.
- Group / shared-resource calendars beyond what the provider exposes through the user's account.
- Free/busy across organizations (Microsoft's `findMeetingTimes` only — no inter-tenant lookups).
- Recurrence-rule editing UX (creating recurring events is in scope; editing a single instance vs. the series is v0.4.1).

## MCP surface

### Tools

- `list_calendars(account_id)` — returns each calendar with `{ id, summary, time_zone, is_primary, access_role }`.
- `list_events(account_id, calendar_id, time_min, time_max, query?)` — paged event list. Returns `{ id, summary, start, end, location, attendees, status, description }`.
- `get_event(account_id, calendar_id, event_id)` — single-event detail with full attendee + conference + attachment info.
- `create_event(account_id, calendar_id, event)` — create. **Approval-gated** under a new `calendar.write` category.
- `update_event(account_id, calendar_id, event_id, patch)` — update. Same approval gate.
- `delete_event(account_id, calendar_id, event_id)` — delete. **Approval-gated** under `calendar.delete`.
- `respond_to_invite(account_id, calendar_id, event_id, response)` — accept / decline / tentative. Approval-gated under `calendar.write`.
- `find_free_slot(account_id, attendees[], duration, time_min, time_max)` — best-effort free-time finder. Read-only.

### Resources (extends v0.3)

- `calendar://account/{id}/{calendar_id}/upcoming?within=24h` — yields events as their start time approaches. Useful for "remind me 10 minutes before each meeting."
- `calendar://account/{id}/invitations/new` — yields newly-arrived meeting invites for triage.

## Permissions

Add two new categories to the per-account permission map:

- `calendar.write` — create, update, RSVP. Default `confirm`.
- `calendar.delete` — delete events. Default `confirm`.

`calendar.read` (list, get, find-free-slot) reuses the `read` category — no UI churn, and read access is symmetric with mail's `read`.

The `draftify` policy doesn't apply to calendar (creating a draft event is the same as creating a tentative private event, which providers don't support uniformly). `draftify` is silently ignored for calendar categories.

## Architecture

```
crates/mail-mcp-core/src/providers/
  gmail/
    mail.rs      (existing)
    calendar.rs  (new — google_calendar API)
    auth.rs      (existing — OAuth scope set extended)
  m365/
    mail.rs      (existing — v0.2)
    calendar.rs  (new — Microsoft Graph /me/calendar*)
    auth.rs      (existing — OAuth scope set extended)
  imap/
    mail.rs      (v0.2)
    /* no calendar — IMAP doesn't have one */
```

OAuth scope additions:
- Google: `https://www.googleapis.com/auth/calendar.readonly` (read), `https://www.googleapis.com/auth/calendar.events` (write)
- Microsoft: `Calendars.Read`, `Calendars.ReadWrite`

Existing accounts get prompted to re-consent on first calendar use (incremental authorization). New accounts gather all scopes during the wizard.

## Wire format

A single `CalendarEvent` struct mirrors Google Calendar's shape (richer than Graph's; we map Graph events into it on read). Time fields are RFC3339 with `time_zone` carried alongside per Google convention. Attendees are `{ email, response_status, optional, organizer }`.

## Tray surface

Settings → Permissions gains two new rows (`calendar.write`, `calendar.delete`). Tray icon's status menu adds an "Upcoming events" submenu showing the next 3 events across all accounts (refreshed every minute), if `calendar.read` allows. Approval dialogs for `calendar.write` show event title + start time + attendee count.

## Testing

- Mock Google Calendar + Graph servers via wiremock.
- Property tests for the Graph→`CalendarEvent` mapping (timezone handling, recurrence rules, attendee lists).
- Permission gate tests: `calendar.write=block` rejects create/update/delete with the standard policy-violation error.
- Approval gate tests: `calendar.write=confirm` parks the create until the user approves.

## Out of scope

- Calendar attachments via Drive / OneDrive. The `attachments` field is surfaced but resolving it to bytes is a follow-up.
- Tasks / To-Do (Google Tasks, Microsoft To Do — separate API surface, separate v0.5+ design).
- Conferencing-link auto-creation. We pass `conferenceData` through verbatim; we don't add Google Meet / Teams links on user-issued create unless the user explicitly asks.
- Full timezone library — we delegate to the system's IANA tz database via `chrono-tz`.
