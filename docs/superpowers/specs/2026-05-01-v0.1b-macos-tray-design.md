# mail-mcp v0.1b — macOS tray app design

## Goals

Ship a polished macOS menu-bar app for managing the mail-mcp daemon: account add/remove, permissions, status, first-run onboarding, approval dialogs, autostart, and signed/notarized distribution. Power users on a NAS or headless box continue to use the v0.1a Rust binaries directly — nothing about that path changes.

## Non-goals

- Windows and Linux tray apps. Each is its own future milestone with its own native stack (WinUI 3, GTK 4).
- Microsoft 365 / IMAP providers. Daemon-side work — orthogonal to the tray.
- App Store distribution. The IPC + spawn-a-daemon pattern is awkward to reconcile with sandboxing; deferred indefinitely.
- Homebrew cask. Useful but additive — DMG comes first.
- M365 OAuth / IMAP forms in the wizard. Wizard supports Gmail only in v0.1b; the wizard is structured so adding providers is local.

## Architecture

```
┌────────────────────────────────────────────────────────┐
│ MailMCP.app (Swift / SwiftUI / AppKit NSStatusItem)    │
│  • Menu bar icon + dropdown                            │
│  • Settings window (Accounts | Permissions | General)  │
│  • First-run wizard window                             │
│  • Approval dialogs (NSAlert modal)                    │
│  • IPC client over UDS                                 │
└──────────────┬─────────────────────────────────────────┘
               │ JSON-RPC over Unix domain socket (newline-framed)
               │ subscribes to approval.requested, account.*, mcp.*
┌──────────────▼─────────────────────────────────────────┐
│ mail-mcp-daemon (existing Rust v0.1a binary)           │
│  Bundled at MailMCP.app/Contents/MacOS/                │
│  (located via Bundle.url(forAuxiliaryExecutable:),     │
│   signed alongside the app)                            │
└────────────────────────────────────────────────────────┘
```

The tray holds **zero persistent state**. Every persistent fact (accounts, permissions, settings flags, MCP pause state) lives in the daemon's SQLite + keychain. The tray re-fetches via IPC whenever its window opens, and stays current via the daemon's notification channel.

The daemon protocol added in v0.1a (`crates/mail-mcp-core/src/ipc/messages.rs` and `proto/ipc-schema.json`) is the contract. No new IPC methods are needed for v0.1b unless gaps surface during implementation.

## IPC client

`IpcClient.swift` mirrors the Rust admin CLI's `IpcClient` (1:1 protocol parity):

- Connect via `Network.framework` to `~/Library/Application Support/mail-mcp/run/ipc.sock` (path resolved via the same `paths.rs` logic ported to Swift, or read from a small JSON config the daemon writes alongside `endpoint.json`).
- One `request(method:params:) async throws -> Value` for unary calls (correlates by JSON-RPC `id`).
- One `subscribe(events:) -> AsyncStream<Notification>` for pub/sub.
- Decoded model types (Account, Permission, PendingApproval, etc.) live in `IPC/Models.swift` and use Codable for symmetry with serde on the Rust side.
- Notifications are decoded into a discriminated `enum Notification` mirroring the Rust `Notification` type.

Failure modes:

- Socket missing → caller surfaces "daemon not running" and triggers `DaemonLauncher.spawn()`.
- Stream EOF → `IpcClient` reconnects with exponential backoff (capped); subscribers see an `AsyncStream` element of `.disconnected` so view-models can show a status banner.
- JSON-RPC error response → thrown as a typed Swift error carrying `code` + `message`.

## Menu-bar UI

`NSStatusItem` with a template (monochrome) icon. The dropdown menu shows:

```
mail-mcp
─────────────────────────────
Status: Connected — 2 accounts
Pending approvals: 0
─────────────────────────────
Open Settings…
Pause MCP
Open MCP Endpoint Info…
─────────────────────────────
About mail-mcp
Quit
```

Status row updates from `StatusViewModel`, which polls `status` every 30s and refreshes immediately on relevant notifications. Pending approvals row shows count and a chevron when > 0; clicking opens the next pending approval in a modal.

## First-run wizard

Auto-opens on app launch when `daemon.status` returns `account_count == 0 && onboarding_complete == false`. Implemented as a single SwiftUI `Window` (not a sheet) with a step controller. Five screens:

1. **Welcome** — short prose: what mail-mcp does, that it stays local. "Continue" / "Skip Setup".
2. **Add first account** — provider grid (Gmail enabled; M365 / IMAP shown disabled with "Coming soon" badge). Pick Gmail → next.
3. **OAuth** — calls `accounts.add_oauth { provider: "gmail" }`, opens system browser via `NSWorkspace.shared.open(url)`. Shows a "Waiting for browser…" overlay with a Cancel button (calls `accounts.cancel_oauth`). On `account.added` notification, advances to step 4. Errors surface inline with a Retry button.
4. **Permissions** — Phase A: read-only display of the v0.1a defaults (Read=Allow, Modify=Allow, Trash=Confirm, Draft=Allow, Send=Draftify) with a one-line explainer per row and a note "you can change these in Settings → Permissions later." `complete_oauth` already calls `install_defaults` daemon-side, so this screen is informational and makes no IPC calls. Phase B optionally upgrades it to interactive popups that mirror the Settings pane.
5. **Autostart** — Yes (recommended) / No radio. "Yes" calls `settings.set_autostart`. Either choice calls `settings.set_onboarding_complete { complete: true }`.
6. **Configure your AI client** — code-block with the Claude Desktop config snippet pre-rendered, "Copy to clipboard" + "Open Claude Desktop config in Finder" actions. "Done" closes the wizard.

The wizard is reachable later from Settings → General → "Run setup again". Re-running skips already-configured steps.

## Settings window

SwiftUI `Settings` scene (gives us native macOS sidebar layout for free). Four panes:

### Accounts pane

- List of accounts: label, email, provider badge, status indicator (connected / needs reauth / network error).
- Per-row actions: "Reauthenticate" (calls `accounts.add_oauth` for re-grant), "Remove" (confirms, calls `accounts.remove`).
- Toolbar: "Add Account…" → reuses the wizard's OAuth screen as a sheet.
- Updates live via `account.added` / `account.removed` / `account.needs_reauth` notifications.

### Permissions pane

- Account picker at the top. Below: a 5-row matrix (Read, Modify, Trash, Draft, Send) × dropdown of policies (Allow, Confirm, Session, Draftify [Send only], Block). Each row has a one-line explainer.
- "Trust this account for this session" button at the bottom (later — calls a `permissions.session_trust` IPC method that doesn't exist yet; if it's not built, defer to v0.1c and remove the button from MVP).
- Calls `permissions.set` on each change (debounced).

### General pane

- Autostart toggle (`SMAppService` + `settings.set_autostart`).
- "Pause MCP" toggle (`mcp.pause`).
- MCP endpoint info card: URL, bearer token (revealed on click, with copy button), stdio-shim path, with copy buttons.
- Log location: read-only path with "Show in Finder" and "Open in Console" buttons.
- "Run setup again" button → re-launches the wizard.

### About pane

- App version, daemon version, daemon uptime.
- Last 50 redacted log lines (read from the daemon's log file via path from General pane). "Copy support bundle" button concatenates version + uptime + log tail to the clipboard.
- Link to GitHub repo + license.

## Approval dialogs

`ApprovalCoordinator` subscribes to `approval.requested` notifications at app launch. For each:

1. Bring the app to focus and present an `NSAlert` modal. Title: `"<Tool> on <account email>"`. Body: a category-specific summary (for Send: from/to/subject/body preview; for Trash: number of messages and a sample of subjects).
2. Buttons: **Approve** / **Reject** / **Approve once** (this session). v0.1b uses two buttons — "Approve" calls `approvals.decide { decision: "approve" }`; "Reject" calls `"reject"`. The session-trust button is deferred (see Permissions pane note above).
3. Auto-resolves on `approval.resolved` notification (e.g., admin CLI decided first) — the modal closes itself.

Modal presentation interrupts the user; this is the right default for v0.1b because the only categories that hit approval flow are Trash and Send (medium-stakes). Read/Modify/Draft default to Allow and never reach the queue.

## Daemon spawning

`DaemonLauncher` runs at app launch:

1. Try `IpcClient.connect()`. If it succeeds, the daemon is already running. Done.
2. If the socket is missing or refuses connections, locate the bundled daemon at `Bundle.main.url(forAuxiliaryExecutable: "mail-mcp-daemon")` and spawn it with `Process` using the env var `MAIL_MCP_GOOGLE_CLIENT_ID` (compiled into the app via Info.plist or fetched from a build-time config — see Build below).
3. Wait up to 5s for `endpoint.json` to appear (the daemon writes it after binding the HTTP server). Reconnect.
4. If startup fails (e.g., the daemon panics), surface a "Daemon failed to start" alert with a "Show logs" button that opens the log directory in Console.app.

The daemon's PID lock (existing `lifecycle::PidLock`) prevents two simultaneous instances. If the user opens the app twice, the second instance just connects to the existing daemon.

## Autostart

`SMAppService.daemon(plistName:)` registers a LaunchAgent that runs the daemon binary at login. The plist is bundled in the app and points at the daemon path inside the bundle. Toggling the General pane switch calls `SMAppService.register()` / `unregister()` and propagates to `settings.set_autostart` so the daemon's status reflects it.

The tray app itself is **not** registered for autostart — only the daemon. The user opens the menu-bar app on demand. (If desired later, a separate "Show in menu bar at login" toggle can register the tray app too.)

## Build system

### Cargo ↔ Xcode

A "Run Script" build phase (`BuildScripts/build-daemon.sh`) runs before the Copy Bundle Resources phase:

```bash
#!/usr/bin/env bash
set -euo pipefail
cd "$SRCROOT/.."   # workspace root
if [ "$CONFIGURATION" = "Release" ]; then
  rustup target add aarch64-apple-darwin x86_64-apple-darwin
  cargo build --release --target aarch64-apple-darwin -p mail-mcp-daemon
  cargo build --release --target x86_64-apple-darwin  -p mail-mcp-daemon
  lipo -create \
    target/aarch64-apple-darwin/release/mail-mcp-daemon \
    target/x86_64-apple-darwin/release/mail-mcp-daemon \
    -output "$BUILT_PRODUCTS_DIR/$EXECUTABLE_FOLDER_PATH/mail-mcp-daemon"
else
  # Debug — host arch only, debug build for fast iteration
  cargo build -p mail-mcp-daemon
  cp "target/debug/mail-mcp-daemon" \
     "$BUILT_PRODUCTS_DIR/$EXECUTABLE_FOLDER_PATH/mail-mcp-daemon"
fi
```

Same script also copies `mail-mcp-stdio` so the AI-client config snippet in the wizard can reference the bundled binary path.

### Google OAuth client_id

The daemon takes `--google-client-id` on the CLI. The tray app needs to pass this when spawning. Two reasonable sources:

- `Info.plist` key `MAIL_MCP_GOOGLE_CLIENT_ID` populated from a build-time `xcconfig` file that's `.gitignore`'d. Each developer drops their own client_id; CI uses a release client_id from secrets.
- The wizard prompts the user for a client_id on first-run if none is configured. (Worse UX for end-users but eliminates the build-time secret problem.)

v0.1b uses the Info.plist + xcconfig approach. The release client_id is owned by the project's Google Cloud Console.

## Distribution

### Signing + notarization

CI is a new `.github/workflows/release.yml` that runs on tag push:

1. Builds universal2 daemon binary via the same `build-daemon.sh` flow.
2. `xcodebuild -scheme MailMCP -configuration Release archive` to produce a `.xcarchive`.
3. `xcodebuild -exportArchive` with a `Developer ID Application` signing identity (cert imported from a base64 secret).
4. `xcrun notarytool submit ... --wait` against an App Store Connect API key.
5. `xcrun stapler staple MailMCP.app`.
6. Builds DMG via `create-dmg` (Homebrew) — bundles the .app + a Sparkle appcast pointer.
7. Uploads DMG + appcast.xml as a GitHub Release asset.

Required GitHub secrets: `MACOS_CERT_P12_BASE64`, `MACOS_CERT_PASSWORD`, `APPLE_API_KEY_BASE64`, `APPLE_API_KEY_ID`, `APPLE_API_ISSUER_ID`, `MAIL_MCP_GOOGLE_CLIENT_ID`, `SPARKLE_PRIVATE_KEY` (Sparkle EdDSA signing).

Until those secrets are populated, releases are unsigned and the install path is "right-click → Open" the first time. The CI workflow is gated behind `${{ secrets.MACOS_CERT_P12_BASE64 != '' }}` so PRs and pre-secret commits don't fail.

### Updates

Sparkle 2 with EdDSA signatures. App checks an appcast hosted on the GitHub Pages branch of the repo (`https://ajthom90.github.io/mail-mcp/appcast.xml`). Update window UI uses Sparkle's defaults.

## Testing

- **Unit (`MailMCPTests`)** — `IpcClient` against a mock UDS server (a Swift coroutine that `accept`s and echoes scripted JSON-RPC responses); decoders for every IPC message type with golden JSON fixtures generated from the Rust `mail-mcp-core` tests.
- **View-model (`MailMCPTests`)** — `AccountsViewModel`, `PermissionsViewModel`, `StatusViewModel` with a fake IPC client. Verify state transitions on each notification type.
- **UI (`MailMCPUITests`)** — happy-path wizard test (XCUITest) launching the app against a real daemon spawned in a tempdir. Smoke-level only.
- **Integration** — extend the existing `crates/mail-mcp-daemon/tests/integration_smoke.rs` ignore-gated test to cover IPC subscribe semantics; the Swift side mocks the daemon for its own tests.

CI matrix: `cargo fmt`/`clippy`/`test` (existing) + a new `xcodebuild test -scheme MailMCP` step gated to macos-14 runners. UI tests run separately on `macos-14` with `-only-testing:MailMCPUITests/WizardSmokeTests`.

## Phased plan

The implementation plan splits into three phases. Each ends with a mergeable, working artifact.

### Phase A — MVP (dogfood-able)

- Xcode project + universal2 build script
- IPC client + Codable models
- `DaemonLauncher` with bundled daemon spawning
- Menu-bar icon + status menu
- First-run wizard (Welcome, Add Account, OAuth, simple Permissions defaults pass-through, Done)
- Accounts pane (list, remove, add via wizard sheet)
- Approval dialogs (modal NSAlert for Trash + Send)
- Unit tests for IPC client and view-models

After Phase A, the maintainer can use the app daily: connect a Gmail account, configure Claude Desktop, observe approvals.

### Phase B — settings completeness

- Permissions pane (full matrix per account)
- General pane (autostart toggle, pause MCP, endpoint info, logs link)
- About pane (version + uptime + redacted log tail + support bundle)
- "Run setup again" entry point
- XCUITest happy-path wizard test

### Phase C — distribution

- xcconfig + Google client_id wiring
- GitHub Actions release workflow (signing, notarization, DMG, Sparkle appcast)
- Sparkle integration in-app
- Install instructions in README
- (Optional, time permitting) Homebrew cask formula

## Open questions deferred to implementation

- **Session-trust button** in Permissions pane — needs a new IPC method `permissions.session_trust { account_id }`. If the implementation reveals it's a 30-line change, do it in Phase B; otherwise defer to v0.1c.
- **Approval dialog body rendering for Send** — full HTML preview (WebKit) vs. plain-text excerpt. Plain-text in Phase A; HTML preview if it becomes a friction point.
- **Universal2 vs. native-arch builds in dev** — script defaults to host-arch in Debug. Revisit if it causes test mismatches.
