# mail-mcp v0.3 — Realtime Events / Push Notifications design

## Goals

Let the daemon proactively surface mail events to the MCP host so models can react to incoming messages without the user having to ask "did anything new arrive?". Examples:

- "Notify me when a reply from Alice lands."
- "Summarize new messages from the priority sender list as they arrive."
- A long-running model task that watches an inbox until a specific reply lands.

In MCP terms, this means exposing per-account event streams to the host through MCP's resources / subscriptions / sampling primitives, fed by the existing daemon notification bus.

## Non-goals

- A new persistent message store. The daemon already keeps a bounded LRU cache; v0.3 adds streaming on top.
- Cross-account aggregation rules ("notify me when ANY account gets a message matching X"). Saved-query routing is v0.5.
- Push-from-cloud (FCM / APNs). Local-only; the daemon polls or uses provider push APIs (Gmail watch, Microsoft Graph subscriptions) and forwards to the MCP host over the existing IPC.
- Background autonomous actions ("auto-archive newsletters"). v0.3 surfaces events; v0.5 introduces user-defined automation.

## Surface area

### MCP resources (new)

Each account exposes a long-lived MCP **resource subscription**:

- `mail://account/{id}/inbox/new` — yields a JSON event per newly-arrived message containing `{ message_id, from, subject, snippet, received_at, labels, is_unread }`. Backed by the daemon's per-provider polling / push.
- `mail://account/{id}/labels/{label}/new` — same but scoped to a label/folder. Useful for INBOX, IMPORTANT, custom labels.
- `mail://account/{id}/threads/{thread_id}/updated` — yields when a thread gets a new message. Useful for "watch this conversation."

Hosts that don't speak MCP subscriptions (some clients are request/response only) keep using the existing tool surface — these resources are additive.

### MCP tools (new)

- `wait_for_reply(account_id, thread_id, timeout_secs)` — blocking call that returns when the watched thread receives a new inbound message OR `timeout_secs` elapses. Built on top of the resource above.
- `wait_for_match(account_id, query, timeout_secs)` — same, but watches a Gmail-style query (the existing `search` query language).

### IPC additions (daemon ↔ tray + daemon ↔ MCP)

Two new daemon notifications already mostly modeled in the existing IPC (extend the tagged enum):

- `mail.new` — `{ account_id, message_id, from, subject, snippet, received_at, labels }`
- `mail.thread_updated` — `{ account_id, thread_id, message_id }`

Tray apps subscribe to surface a "you have new mail in account X" indicator alongside the approval count.

## Architecture

```
┌─────────────────────────────────────────────────────────┐
│ MCP host (Claude Desktop, Code, etc.)                   │
│  Subscribes to mail://account/{id}/inbox/new            │
│  Calls wait_for_reply / wait_for_match                  │
└──────────────┬──────────────────────────────────────────┘
               │ MCP resources/subscriptions, tool calls
┌──────────────▼──────────────────────────────────────────┐
│ mail-mcp-daemon                                         │
│  • EventBus<Event> — broadcast::channel<Event>          │
│  • Per-account "watchers" (provider-specific)           │
│      Gmail: gmail.users.watch + Pub/Sub OR poll fallback│
│      Graph: subscriptions endpoint OR poll fallback     │
│  • IPC + MCP both subscribe to the bus                  │
└─────────────────────────────────────────────────────────┘
```

The bus is the same primitive that already powers the v0.1a/b approval / status notifications; v0.3 adds new event variants and the providers that emit them.

### Provider watchers

**Gmail** (`crates/mail-mcp-core/src/providers/gmail/watcher.rs`)
- Preferred: `users.watch()` + Cloud Pub/Sub push subscription. Requires the user to have a GCP project; we'll degrade gracefully when not configured.
- Fallback: poll `history.list` every 30 s with the previous `historyId`. Bandwidth-friendly because Gmail returns just the diff.

**Microsoft Graph** (`crates/mail-mcp-core/src/providers/m365/watcher.rs`)
- Preferred: `subscriptions` endpoint with a webhook URL. The daemon's local HTTP listener doesn't accept inbound from the public internet, so we tunnel via... [TBD: revisit when v0.3 actually starts; might be poll-only on initial release].
- Fallback: poll `/messages/delta` every 30 s.

### Coalescing & rate limiting

A burst of mail arriving (e.g., a mailing-list digest) shouldn't spam the MCP host. The bus debounces same-thread events within 5 s and caps cross-thread fan-out at 10 events/sec/account; excess gets a single coalesced "N more new messages" event.

## Security / approval

The new resources don't touch the approval gate — they're read-only event streams scoped by per-account `read` policy. If `read=block` for an account, the resource simply yields nothing for that account (and `wait_for_*` returns "blocked by policy" immediately).

## Out of scope

- Attachment streaming via these events (v0.3 events carry only metadata; attachments are fetched on demand via the existing tool path).
- User-defined event filters beyond `query`. v0.3 ships query+thread+label filters; richer rule-engine semantics are v0.5.
- Cross-MCP-host fan-out (one daemon → multiple hosts). v0.3 supports it because it's already how the bus works, but UX-wise the tray surfaces only one host's connection state.

## Testing

- Mock provider backend that emits scripted events into the watcher → assert events flow through the bus → MCP resource → MCP host.
- Property test for coalescing: random burst patterns, verify the rate limit is honored and no events are dropped without a coalesced replacement.
- Tray smoke: receive a `mail.new` notification and confirm the menu's "new messages" count updates.
