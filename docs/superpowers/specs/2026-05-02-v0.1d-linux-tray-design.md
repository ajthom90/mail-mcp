# mail-mcp v0.1d — Linux tray app design

## Goals

Ship a polished Linux tray (system indicator) app for the mail-mcp daemon, feature parity with the v0.1b macOS / v0.1c Windows trays. Power users on a NAS / headless server keep using the v0.1a Rust binaries directly — nothing changes for them.

## Non-goals

- Per-distro packaging (.deb, .rpm, AUR). v0.1d ships a tarball + Flatpak; distro packagers can pick it up later.
- Snap. Flatpak is the modern cross-distro path; Snap can come later if requested.
- Wayland-specific protocols beyond what GTK4 + libadwaita provide out of the box.
- KDE-native widget. The GTK4/libadwaita app runs fine under KDE Plasma but doesn't use Kirigami.

## Architecture

```
┌────────────────────────────────────────────────────────┐
│ mail-mcp-tray (C, GTK4 + libadwaita + AyatanaAppIndicator) │
│  • System tray icon (AyatanaAppIndicator3)             │
│  • Settings window (AdwPreferencesWindow)              │
│  • First-run wizard (AdwAssistant)                     │
│  • Approval dialogs (AdwMessageDialog)                 │
│  • IPC client over Unix domain sockets                 │
└──────────────┬─────────────────────────────────────────┘
               │ JSON-RPC over UDS (newline-delimited)
               │ ${XDG_RUNTIME_DIR}/mail-mcp/ipc.sock
┌──────────────▼─────────────────────────────────────────┐
│ mail-mcp-daemon (existing Rust binary)                 │
└────────────────────────────────────────────────────────┘
```

The daemon's UDS-based IPC (the original `mail-mcp-core::ipc::transport::IpcListener` Unix path) needs no changes. Linux uses the exact same socket path the macOS tray already uses — the only difference is `XDG_RUNTIME_DIR` placement (`/run/user/<uid>/mail-mcp/ipc.sock`) instead of the macOS `${TMPDIR}` variant.

## Why C + GTK4 + libadwaita (vs. Rust + gtk-rs)

Considered alternatives:
- **Rust + gtk4-rs / libadwaita-rs** — keeps everything Rust. Crate ecosystem is stable but has a perpetual catch-up to the C API; some libadwaita widgets land in -rs months after C. Build complexity adds up (cargo + system gobject-introspection).
- **Python + PyGObject** — fastest to write, but the Python interpreter ships in the bundle and you spend lots of effort packaging.
- **Vala** — direct compile to GTK C, but Vala's tooling is sparser than C's and few maintainers know it.
- **Tauri** — pulls webkitgtk and a webview chrome; bundle size 60+ MB.
- **Avalonia** — cross-platform, but feels foreign on Linux and uses .NET.
- **Native Qt + KF6** — KDE-native but heavyweight on GNOME.

C with GTK4 + libadwaita gives us native widgets on every distro that ships GNOME / GTK, modest binary size, dependency on system libraries that distros expect to update independently of the app, and good Wayland/X11 coverage.

## IPC client

`ipc_client.c` uses `socket(AF_UNIX, SOCK_STREAM, 0)` + `connect()` (the v0.1a-IPC fix in PR #11 used the same primitives in Swift; the same approach in C is straightforward). I/O is line-based: read into a buffer, scan for `\n`, hand each frame to a callback.

Three async modes:
1. `ipc_call_sync(client, method, params, ...)` — synchronous JSON-RPC; blocks until response.
2. `ipc_call_async(client, method, params, callback)` — non-blocking; the callback fires on the GTK main loop when the response arrives.
3. `ipc_subscribe(client, events, on_notification)` — registers a per-event-type callback. The internal poll loop calls the callbacks via `g_main_context_invoke` so they run on the UI thread.

The IPC client owns a `GSource` that integrates with `GMainContext`, so reads happen as part of GTK's event loop without spawning a separate thread. This matches the GLib idiom and avoids cross-thread invocation gymnastics.

JSON parsing uses `json-glib` (already a libadwaita dependency tree member, no extra system dep). Method/param tagging follows the v0.1a IPC schema verbatim.

## Tray icon

`AyatanaAppIndicator3` (the modern fork of the original `libappindicator`) gives us a tray icon that works on GNOME (via gnome-shell-extension-appindicator), KDE (native), XFCE, MATE, and most other DEs. We bundle a monochrome SVG from the resource bundle. Menu structure mirrors macOS:

```
MailMCP
─────────────────────────
Status: Connected — 2 accounts
Pending approvals: 0
─────────────────────────
Open Settings…
Pause MCP
Open MCP Endpoint Info…
─────────────────────────
Run Setup Again…
Check for Updates…
─────────────────────────
Quit
```

For desktops that don't show app indicators (e.g. vanilla GNOME without the extension), we surface a clear setup hint in the README and ship a fallback `gnome-shell` notification on first launch ("To see the MailMCP tray icon, install the AppIndicator extension").

## First-run wizard

`AdwAssistant` (assistant pattern matching the macOS NSWindow + step controller) hosts six pages: Welcome → Add Account → OAuth → Permissions → Autostart → Configure AI client. Same logical flow as macOS / Windows.

OAuth: `gtk_show_uri(window, auth_url, GDK_CURRENT_TIME)` opens the system browser. The wizard listens for `account.added` notifications and advances when the daemon reports the new account.

Autostart: writes a systemd user unit (`~/.config/systemd/user/mail-mcp-daemon.service`) and enables it via `systemctl --user enable mail-mcp-daemon.service`. Falls back to a `~/.config/autostart/io.github.ajthom90.MailMCP.desktop` file if systemd-user isn't available (e.g., some non-systemd distros, though those are rare in 2026).

## Settings window

`AdwPreferencesWindow` hosts four `AdwPreferencesPage`s (Accounts / Permissions / General / About). Each page is loaded from a `.ui` resource. Layout matches v0.1b/v0.1c.

## Approval dialogs

`AdwMessageDialog` for each approval. Subscribes to `approval.requested` notifications, presents one dialog per pending approval, dispatches `approvals.decide` based on the button clicked.

Dialogs are non-modal but raised above the rest of the desktop (urgency hint set). On Wayland we can't truly force-focus, but raising + flashing the indicator icon is the convention.

## Daemon spawning

Standard pattern:

1. Try `connect()` to `${XDG_RUNTIME_DIR}/mail-mcp/ipc.sock`.
2. If `ENOENT` or `ECONNREFUSED`, locate the daemon at one of:
   - `${MEson installation dir}/bin/mail-mcp-daemon` (system / Flatpak)
   - `${XDG_DATA_HOME}/mail-mcp/bin/mail-mcp-daemon` (user-local install)
   - PATH lookup as fallback
3. `posix_spawn` with `--google-client-id` env-overridable.
4. Wait up to 5 s for `endpoint.json` to appear.
5. Reconnect.

The daemon's `fs2` PID lock keeps duplicates at bay.

## Autostart

systemd user unit:

```ini
# ~/.config/systemd/user/mail-mcp-daemon.service
[Unit]
Description=mail-mcp daemon

[Service]
ExecStart=%h/.local/bin/mail-mcp-daemon --google-client-id ${MAIL_MCP_GOOGLE_CLIENT_ID}
Restart=on-failure

[Install]
WantedBy=default.target
```

A toggle in Settings → General writes / removes the file and runs `systemctl --user {enable,disable} mail-mcp-daemon.service`. The tray app itself is NOT autostarted — only the daemon. The user opens the tray app on demand.

## Build system

### Meson + cargo

`meson.build` declares the GTK4 + libadwaita + ayatana-appindicator + json-glib dependencies. A `custom_target` invokes `BuildScripts/build-daemon.sh` to produce `mail-mcp-daemon` for the host architecture (`x86_64-unknown-linux-gnu` or `aarch64-unknown-linux-gnu`); the resulting ELF is installed to `${prefix}/bin/`.

`tray-app-linux/BuildScripts/build-daemon.sh` invokes `cargo build --release -p mail-mcp-daemon`. Same shape as the macOS version, simpler (no lipo, no codesign).

### Google OAuth client_id

A Meson option (`-Dgoogle_client_id=<your-id>`) is baked into a generated `config.h` header at build time. Distro packagers and end users can override at install or runtime via `MAIL_MCP_GOOGLE_CLIENT_ID` env var.

## Distribution

### Tarball + binary release

`meson dist` produces a source tarball; the release CI workflow runs `meson setup build --buildtype=release && meson compile -C build` on `ubuntu-latest` and packages the resulting binaries into a `MailMCP-<version>-x86_64-linux.tar.gz` (and aarch64 via cross-compile). End users `tar -xf` and run from the unpack dir.

### Flatpak

`io.github.ajthom90.MailMCP.yml` is a `flatpak-builder` manifest that:
- Includes `org.gnome.Platform//46` as the runtime
- Adds the daemon binary as a sub-module that builds via cargo
- Bundles the AppStream metadata so it shows up in GNOME Software / KDE Discover

CI builds the Flatpak nightly via `flatpak-github-actions`. Publishing to Flathub is a follow-up — the manifest in-repo is enough for users to `flatpak-builder` locally.

### AppImage

Optional v0.1d.1 follow-up. AppImages are nicer for users who don't want Flatpak but the build path is finicky.

### Code signing

Linux desktop apps are conventionally not signed. Flatpak bundles get signed by Flathub if accepted. Tarballs can include a `.sig` file (GPG) if the maintainer wants — out of scope for v0.1d MVP.

## Testing

- **Unit (CMocka)** — `models.c`, `paths.c`, `ipc_client.c` (against a hand-rolled mock UDS server using a child thread).
- **Integration** — extend `crates/mail-mcp-daemon/tests/integration_smoke.rs` with a Linux-only path that pokes the UDS directly. (No GUI integration test in v0.1d; the wizard happy-path is manual smoke for now.)
- **CI** — new `linux-tray` job on `ubuntu-latest` that installs `meson`, `gcc`, `libgtk-4-dev`, `libadwaita-1-dev`, `libayatana-appindicator3-dev`, `libjson-glib-dev`, and runs `meson test`.

## Phased plan

Three phases, mirroring v0.1b/v0.1c:

### Phase A — MVP (dogfood)

- Meson project skeleton + build-daemon.sh + main.c + AdwApplication boilerplate
- IpcClient + Codable models
- DaemonLauncher
- Tray icon + status menu
- First-run wizard (Welcome, Add Account, OAuth, Permissions defaults pass-through, Autostart, Configure AI client)

- Approval dialogs

### Phase B — settings completeness

- Permissions page (full matrix)
- General page (autostart toggle, pause, endpoint info, logs)

### Phase C — distribution

- AppStream metadata
- systemd user unit + autostart toggle

- Tarball release CI
