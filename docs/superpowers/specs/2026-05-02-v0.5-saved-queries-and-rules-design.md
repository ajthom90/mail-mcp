# mail-mcp v0.5 — Saved queries + automation rules design

## Goals

Two related capabilities:

1. **Saved queries** — let the user (or a model on the user's behalf) name and store a search query. Saved queries become first-class IPC + MCP citizens: they show up in the tray, get their own MCP resource, and can be referenced by name from tools (`search(saved="vip")` instead of pasting the same complex query string).
2. **Automation rules** — let the user (and only the user — never a model unsupervised) define rules of the form `when <trigger> matches <query>, do <action>`. Triggers and actions are constrained to the existing tool surface; rules run inside the daemon under the existing per-account permission gate. This is what turns "watch for X" (v0.3 push events) into "watch for X and do Y."

Together these let a power user say "auto-archive newsletters older than 30 days" or "summarize daily what came in to my Action Required label" without each model session re-deriving the rule.

## Non-goals

- Cross-account rule conditions in v0.5. A rule applies to one account.
- Rule-driven sends without a confirm step. v0.5 ALWAYS gates send-class actions through the `send` permission policy (default `confirm`); the rule engine is just a scheduled tool-call dispatcher.
- A DSL with branching / loops. Rules are linear `match → action[]` lists.
- Cloud sync for saved queries / rules. They live in `${XDG_CONFIG_HOME}/mail-mcp/`. Backup / sync is the user's problem.
- Model-authored rule install. Rules are user-installed only via the tray UI or admin CLI.

## Saved queries

### Storage

`${XDG_CONFIG_HOME}/mail-mcp/saved-queries.toml`:

```toml
[[query]]
name = "vip"
account = "01H123"               # or "*" for all accounts
expression = "from:(alice@... OR bob@...) AND newer_than:7d"
description = "VIP senders, last week"

[[query]]
name = "action-required"
account = "*"
expression = "label:action-required AND is:unread"
description = "Open action-required mail across all accounts"
```

Loaded at daemon startup; reloaded on `SIGUSR1` or via the `queries.reload` admin RPC.

### Surface

- IPC: `queries.list`, `queries.get`, `queries.add`, `queries.remove`, `queries.update`. Tray binds a Settings → Saved Queries page to these.
- MCP tool: `search(account_id, saved=<name>)` — runs the named query. The existing `search(account_id, query=<expr>)` keeps working.
- MCP resource (extends v0.3): `mail://account/{id}/saved/{name}/new` — yields when a new message matches the saved query.

## Automation rules

### Storage

`${XDG_CONFIG_HOME}/mail-mcp/rules.toml`:

```toml
[[rule]]
name = "auto-archive newsletters"
account = "01H123"
trigger = "mail.new"             # one of: mail.new, calendar.new_invite, schedule.daily, schedule.hourly
match = "list-id:newsletters"    # query expression; required for mail.new triggers
actions = [
  { tool = "modify_labels", args = { remove = ["INBOX"], add = ["archive"] } },
]
enabled = true

[[rule]]
name = "daily action-required digest"
account = "*"
trigger = "schedule.daily"
schedule_time = "08:00"          # local time
actions = [
  { tool = "search", args = { saved = "action-required" } },
  { tool = "summarize", args = { input_from = "previous_step" } },
  { tool = "send_message", args = { to = ["self"], subject = "Daily AR digest", body_from = "previous_step" } },
]
enabled = true
```

(Note: `summarize` and the `previous_step` chaining are sketched; final shape needs more design. v0.5 may ship rules WITHOUT chaining and add it in v0.5.1.)

### Trigger types

- `mail.new` — fires per matching new message. Action receives the message id.
- `calendar.new_invite` — fires per matching new invite. Action receives event id.
- `schedule.hourly` / `schedule.daily` — fires on cron-like schedule. No `match` expression; the action gets no per-event input.

### Action set

Initially restricted to a safe-list of read + label-modify + archive + draft-create tools. `send_message`, `delete_message`, `delete_event` are allowed but ALWAYS gated by the per-account permission policy (regardless of the rule firing). I.e., a rule can ENQUEUE a send, but if `send=confirm`, the user still gets prompted before it goes out.

### Approval semantics

Rules don't bypass approvals. If a rule's action is gated, the daemon parks the action and emits `approval.requested` exactly as if the model had called the tool directly. The `approval.requested` payload gets a `triggered_by_rule` field so the tray can show "Rule '<name>' wants to send to <to>" instead of "MCP host wants to send".

### Execution model

The daemon spawns one worker task per rule. Workers subscribe to the EventBus (v0.3) for trigger events and to a cron source for scheduled triggers. Failures (transient network, rate limits) get exponential-backoff retry with a circuit breaker after 5 consecutive failures.

### Audit log

Every rule firing writes a line to `${XDG_DATA_HOME}/mail-mcp/rules.log` with timestamp, rule name, trigger payload, actions taken, outcomes. The tray's Settings → Rules page surfaces the last 100 firings per rule.

## Tray surface

- Settings → Saved Queries: list / add / edit / remove. Test-run button shows result count.
- Settings → Automation Rules: list / add / edit / remove / toggle. "Last run" + "Last error" + "Show audit log" per rule.
- Status menu badge: "3 rules active" / "Rule X errored — click to inspect."

## Security

Rules are user-only. The MCP host has no API to install or modify rules. The IPC `rules.add` / `rules.update` / `rules.remove` calls require the local-only privileged channel — same authentication as the existing `accounts.add_oauth`. The remote MCP host's tool surface intentionally doesn't expose them.

## Testing

- Property tests for the rule engine: random rule sets + random event streams, verify exactly the matching events fire actions.
- Approval-gating: a rule's action under `send=confirm` produces `approval.requested` with `triggered_by_rule` set.
- Schedule trigger: clock-mocking to verify daily / hourly fire on the right ticks.

## Out of scope

- Rule chaining (`previous_step` reference). v0.5 ships single-action rules; chaining is v0.5.1.
- Rule sharing / templates. v0.5.2 maybe.
- A visual rule builder. v0.5 ships TOML editing via the tray's "Open in editor" button; a real builder UI waits.
