# mail-mcp v0.2 — Additional providers (M365 + IMAP)

## Goals

Make mail-mcp useful for users who don't live in Gmail. v0.2 adds two providers behind the existing `MailProvider` trait:

1. **Microsoft 365 / Outlook** via Microsoft Graph API + OAuth 2.0 PKCE.
2. **Generic IMAP/SMTP** for any provider that exposes IMAP (Fastmail, ProtonMail Bridge, self-hosted Dovecot, etc.).

Both work on every platform that v0.1a/b/c support — no UI changes beyond enabling the previously-disabled-with-"Coming soon" provider buttons in the wizard's Add Account step.

## Non-goals

- Microsoft Exchange On-Premises (EWS). The Graph API path covers M365 cloud only; on-prem EWS would need a separate provider with different auth and a different envelope shape.
- POP3. IMAP-only.
- M365 calendar / contacts / Teams. Mail only.
- IMAP IDLE-driven push. v0.2 polls; IDLE can come later.
- iCloud Mail via IMAP. The IMAP code path supports it in principle, but Apple's app-specific-password flow is friction enough that we explicitly call it out as untested in v0.2.

## Architecture

Both providers slot in alongside Gmail under `crates/mail-mcp-core/src/providers/`:

```
crates/mail-mcp-core/src/providers/
  gmail/                 # v0.1a — unchanged
  m365/                  # NEW
    mod.rs
    client.rs            # auth-aware Graph HTTP client
    provider.rs          # MailProvider trait impl
    messages.rs          # search + threads + messages
    folders.rs           # folder listing (M365 has folders, not labels)
    categories.rs        # M365 categories ≈ Gmail labels (additive tags)
    triage.rs            # mark_read / star (flag) / categorize / move / archive
    compose.rs           # drafts + send via Graph
    parse.rs             # decode Graph message JSON → domain Message
  imap/                  # NEW
    mod.rs
    client.rs            # async-imap session pool + reconnect
    smtp.rs              # outgoing via lettre
    provider.rs          # MailProvider trait impl
    messages.rs          # SEARCH / FETCH / store keywords
    folders.rs           # mailbox tree
    triage.rs            # \Seen / \Flagged / keywords / MOVE
    compose.rs           # APPEND to Drafts / SMTP send
    parse.rs             # mail-parser → domain Message
```

`mail-mcp-daemon` gets a `provider_for_account` shim that picks the right impl based on `Account.provider` (already a `ProviderKind` enum: Gmail, M365, Imap).

The `accounts.add_oauth` IPC method gains `"m365"` as a valid provider value. A new `accounts.add_imap` method takes a typed config (server, port, username, password, TLS mode).

## OAuth: Microsoft Graph

### Endpoints

| Endpoint | URL |
|---|---|
| auth_url | `https://login.microsoftonline.com/common/oauth2/v2.0/authorize` |
| token_url | `https://login.microsoftonline.com/common/oauth2/v2.0/token` |
| user_info | `https://graph.microsoft.com/v1.0/me` |

`/common` (vs `/{tenant}`) accepts both consumer and AAD accounts. Personal accounts (outlook.com, hotmail.com) and work/school accounts both work — the user picks during the consent prompt.

### Scopes

```
offline_access                   # required for refresh_token
Mail.ReadWrite                   # read + modify (label, mark read, move)
Mail.Send                        # send (subject to user's send-policy)
User.Read                        # for /me endpoint to learn the user's address
```

`MailboxSettings.Read` is NOT required (we don't read out-of-office settings or signatures yet).

### PKCE

Same flow as Gmail. `mail-mcp-core::oauth::PkcePair::generate()` already produces a 64-char verifier + S256 challenge that meets Microsoft's requirements.

### Differences vs Gmail

- Microsoft refresh tokens **rotate on every refresh** (unlike Google's stable tokens). The `RefreshRotationCallback` from PR #10 (issue #2 fix) handles this — Microsoft hits that path on every refresh, where Google rarely does.
- Microsoft access tokens are JWTs; we don't decode them, just send as Bearer.
- Token responses include `id_token` for OIDC; we ignore it (we use `/me` instead).

### Provider config

```rust
// crates/mail-mcp-core/src/oauth/microsoft.rs
pub fn config(client_id: impl Into<String>) -> ProviderConfig {
    ProviderConfig {
        auth_url:    "https://login.microsoftonline.com/common/oauth2/v2.0/authorize".into(),
        token_url:   "https://login.microsoftonline.com/common/oauth2/v2.0/token".into(),
        client_id:   client_id.into(),
        default_scopes: vec![
            "offline_access".into(),
            "https://graph.microsoft.com/Mail.ReadWrite".into(),
            "https://graph.microsoft.com/Mail.Send".into(),
            "https://graph.microsoft.com/User.Read".into(),
        ],
    }
}
```

## M365 provider via Graph API

### Domain model mapping

| mail-mcp domain | Graph |
|---|---|
| `Message` | `/me/messages/{id}` |
| `Thread` | `/me/conversations/{id}` (Graph threads) |
| `Folder` | `mailFolder` resource (INBOX = `inbox`, SENT = `sentitems`, TRASH = `deleteditems`, DRAFT = `drafts`) |
| `Label` | Graph "categories" (additive tags) — `Message.categories: string[]` |
| `Draft` | `/me/messages` with `isDraft: true` |

### Endpoint mapping

| Trait method | Graph call |
|---|---|
| `search(q)` | `GET /me/messages?$search="<q>"&$top=<limit>&$skiptoken=<cursor>` (search uses KQL) |
| `get_thread(id)` | `GET /me/conversations/{id}/messages?$expand=...` |
| `get_message(id)` | `GET /me/messages/{id}` |
| `list_folders()` | `GET /me/mailFolders?$top=100` (filter to system folders by `wellKnownName`) |
| `list_labels()` | `GET /me/outlook/masterCategories` |
| `list_drafts()` | `GET /me/mailFolders/drafts/messages` |
| `mark_read(ids, true)` | `PATCH /me/messages/{id}` body `{"isRead": true}` |
| `star(ids, true)` | `PATCH /me/messages/{id}` body `{"flag": {"flagStatus": "flagged"}}` |
| `label(ids, l, true)` | `PATCH /me/messages/{id}` body `{"categories": [...]}` (additive) |
| `move_to(ids, folder)` | `POST /me/messages/{id}/move` body `{"destinationId": "<folderId>"}` |
| `archive(ids)` | `POST /me/messages/{id}/move` to `archive` well-known folder |
| `trash(ids)` | `POST /me/messages/{id}/move` to `deleteditems` |
| `untrash(ids)` | `POST /me/messages/{id}/move` to `inbox` (or original folder if trackable) |
| `create_draft(d)` | `POST /me/messages` with `isDraft: true` |
| `update_draft(id, d)` | `PATCH /me/messages/{id}` |
| `send_message(m)` | `POST /me/sendMail` (creates + sends in one call) |
| `send_draft(id)` | `POST /me/messages/{id}/send` |

### Pagination

Graph uses `@odata.nextLink` URLs in responses. The provider stores the full nextLink as the opaque `cursor` field of `SearchQuery`/`SearchResults`.

### Throttling

Graph applies per-app and per-user throttling (HTTP 429 with a `Retry-After` header). The shared `AuthClient` already retries on 401; add a 429 path that respects `Retry-After` (capped at 30 s, max 3 retries). This logic lives in `mail-mcp-core::providers::common` so Gmail benefits too (Gmail returns 429 for quota exhaustion).

## IMAP/SMTP provider

### Crate choice

- **`async-imap` 0.10+** — read/triage. Built on `async-std`-or-`tokio`; we use the `tokio` feature.
- **`mail-parser` 0.9+** — already in workspace, parses raw RFC 5322 into our domain `Message`.
- **`lettre` 0.11+** — outgoing SMTP with TLS, async-std-or-tokio (we use tokio). Supports STARTTLS + implicit TLS, OAUTHBEARER, and PLAIN.

### Configuration

```rust
pub struct ImapConfig {
    // Connection
    pub imap_host: String,
    pub imap_port: u16,                     // 993 (TLS) / 143 (STARTTLS)
    pub imap_security: TlsMode,
    pub smtp_host: String,
    pub smtp_port: u16,                     // 465 (TLS) / 587 (STARTTLS)
    pub smtp_security: TlsMode,
    // Auth
    pub username: String,
    // Password lives in the OS keychain alongside refresh tokens.
}

pub enum TlsMode { ImplicitTls, StartTls }
```

The wizard's "Add IMAP" form collects these. Fastmail/ProtonMail-bridge get auto-detected presets.

### Trait method mapping

| Trait method | IMAP / SMTP |
|---|---|
| `search(q)` | `SEARCH X-GM-RAW <q>` if Gmail-IMAP, else `SEARCH BODY <q>` |
| `get_thread(id)` | Re-implement threading via References/In-Reply-To headers (no IMAP THREADS extension assumed) |
| `get_message(id)` | `FETCH <uid> (BODY[] FLAGS)` then `mail-parser::Message::parse` |
| `list_folders()` | `LIST "" "*"`, filter by SPECIAL-USE flags or RFC 6154 attributes |
| `list_labels()` | Server keywords from `LIST` response — IMAP has no real labels |
| `mark_read(ids, true)` | `STORE <uids> +FLAGS (\Seen)` |
| `star(ids, true)` | `STORE <uids> +FLAGS (\Flagged)` |
| `label(ids, l, on)` | `STORE <uids> +FLAGS ($Foo)` (custom keyword) |
| `move_to(ids, folder)` | `MOVE <uids> <folder>` (RFC 6851) or COPY+EXPUNGE fallback |
| `archive(ids)` | MOVE to `\Archive`-attributed folder |
| `trash(ids)` | MOVE to `\Trash`-attributed folder |
| `untrash(ids)` | MOVE to INBOX (no preserved-original-folder concept in IMAP) |
| `create_draft(d)` | Build MIME via `mail-builder` (already in workspace) → `APPEND <Drafts> {flags=(\Draft)}` |
| `update_draft(id, d)` | APPEND new + EXPUNGE old (IMAP can't edit in place) |
| `send_message(m)` | `lettre::AsyncSmtpTransport::send` |
| `send_draft(id)` | FETCH the draft, `lettre::send`, then `STORE +FLAGS (\Deleted)` + EXPUNGE |

### Connection management

A single long-lived IMAP connection per account, held inside the provider. On disconnect (any I/O error) reconnect lazily on the next call. SMTP connections are short-lived: open, send, close.

### Quirks worth calling out

- **Gmail IMAP** has weird folder semantics ("[Gmail]/All Mail", "[Gmail]/Sent Mail"). Hard-code recognition of those.
- **ProtonMail Bridge** runs on `127.0.0.1:1143` / `127.0.0.1:1025` with self-signed certs — the connection layer needs a per-account "trust this cert fingerprint" override.
- **Fastmail** uses `imap.fastmail.com:993` with regular passwords or app-specific passwords. Their JMAP API is faster but out of scope.
- **iCloud Mail** requires app-specific passwords. We surface a clear error message if PLAIN auth fails with "Application-specific password required."

## Permissions integration

No changes. The `MailProvider` trait is provider-agnostic; the daemon's permission enforcement (`mail-mcp-core::permissions::enforce`) already applies uniformly. Send-policy = Draftify on M365 → `convert_send_to_draft` → `POST /me/messages` with `isDraft: true` (instead of `/me/sendMail`).

## Wizard / Settings UX

- Add Account screens for M365 and IMAP move from "Coming soon" to enabled.
- M365 OAuth screen reuses the same `accounts.add_oauth` IPC method with `provider: "m365"`.
- IMAP form is a new screen (host/port/username/password/TLS) and a new `accounts.add_imap` IPC method that constructs an `ImapConfig`, stores the password in the keychain, and returns the new Account.
- Settings → Permissions per-account matrix is unchanged.

## Phased plan

Three independent sub-milestones; each is its own PR.

### v0.2.0 — M365 provider
- `oauth::microsoft` module with PKCE config
- `providers::m365::*` (8-10 files mirroring Gmail's structure)
- Wizard: enable M365 button, route OAuth through the existing `add_oauth` flow
- ~1200-1600 LOC + tests

### v0.2.1 — IMAP/SMTP provider
- `providers::imap::*` (8-10 files)
- New `accounts.add_imap` IPC method + IMAP config form in wizard
- New IMAP-specific tests using `wiremock`-style imap mock crate (or hand-rolled with `tokio::net::TcpListener`)
- ~1500-1800 LOC + tests

### v0.2.2 — Throttling + reconnect polish
- Shared 429/Retry-After helper used by both Gmail and M365
- IMAP reconnect with exponential backoff
- IMAP IDLE for push notifications (optional; can defer to v0.3)

## Testing strategy

- **Unit**: `wiremock` for Graph API, hand-rolled IMAP mock for IMAP (tokio TCP listener + canned protocol responses). Same TDD pattern as the Gmail provider.
- **Integration**: opt-in tests against a real M365 dev-tenant account (gated by env vars; only the maintainer runs them locally).
- **CI**: extend the existing matrix; M365 + IMAP build on every supported target without new system deps.

## Open questions deferred to implementation

- **iCloud Mail** support — it's just IMAP, but Apple's app-password UX is friction; a follow-up sub-milestone could add a "Connect iCloud Mail" wizard page that walks the user through generating an app-specific password.
- **OAUTHBEARER for IMAP** (RFC 7628) — Gmail-IMAP and M365-IMAP both support OAuth-bearer-token IMAP login. v0.2 starts with PLAIN; OAUTHBEARER is a v0.2.1 nice-to-have.
- **JMAP** as a parallel path for Fastmail and Apple Mail (when iOS 18 lands) — would be a v0.3 milestone.
- **MIME signature detection** — strip "Sent from my iPhone" / signature blocks before snippet generation? Probably no (snippets are short anyway).
- **Cross-account thread merge** — when the same thread exists in two accounts, do we de-dup? Probably no for v0.2.

## Out of scope (future milestones)

- Exchange On-Premises (EWS) — own milestone.
- POP3 — won't.
- Microsoft Exchange Web Services (EWS) — won't (Graph subsumes).
- IMAP push via NOTIFY — niche, deferred.
- Mail rules / filters management.
