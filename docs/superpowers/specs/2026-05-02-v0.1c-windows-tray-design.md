# mail-mcp v0.1c — Windows tray app design

## Goals

Ship a polished Windows menu-bar (system tray) app for managing the mail-mcp daemon. Feature parity with the v0.1b macOS tray: account add/remove, permissions, status, first-run wizard, approval dialogs, autostart, signed installer distribution. Headless / NAS users keep using the v0.1a Rust binaries directly — nothing about that path changes.

## Non-goals

- Microsoft Store distribution. The Windows packaging path uses MSIX-via-Wix or a plain MSI; Microsoft Store has tighter sandbox constraints that would force an IPC redesign.
- Linux tray app — its own milestone (v0.1d).
- M365 / IMAP providers — v0.2.
- Per-account IPC encryption (named pipes default to per-user ACLs which are sufficient on a typical workstation).

## Architecture

```
┌────────────────────────────────────────────────────────┐
│ MailMCP.exe (.NET 8, WinUI 3 + Windows App SDK)        │
│  • System tray icon (NotifyIcon via WinUI)             │
│  • Settings window (NavigationView pages)              │
│  • First-run wizard window                             │
│  • Approval dialogs (ContentDialog)                    │
│  • IPC client over named pipes                         │
└──────────────┬─────────────────────────────────────────┘
               │ JSON-RPC over named pipe (newline-framed)
               │ \\.\pipe\mail-mcp-<USERNAME>
               │ subscribes to approval.requested, account.*, mcp.*
┌──────────────▼─────────────────────────────────────────┐
│ mail-mcp-daemon.exe (existing Rust binary)             │
│  Bundled at <install>\mail-mcp-daemon.exe              │
└────────────────────────────────────────────────────────┘
```

The daemon's named-pipe IPC backend (`mail-mcp-core::ipc::transport`, shipped in PR #11) is the contract. No new IPC methods are needed for v0.1c unless gaps surface during implementation.

## Why WinUI 3 + .NET

Considered alternatives:
- **WPF** — mature, well-documented, but unfashionable and lacks Mica/acrylic surfaces.
- **WinForms** — fastest to write, ugliest result.
- **WinUI 2 / UWP** — sandboxed, IPC + spawn-a-daemon awkward.
- **Tauri** — Rust-native, but pulls in WebView2; bundle size ~100 MB.
- **Avalonia** — cross-platform, but the v0.1d Linux tray plan calls for native GTK4 anyway.
- **Native Win32 + COM** — no.

WinUI 3 + .NET 8 gives us native Fluent UI, modest binary size (~30 MB self-contained), and clean P/Invoke for tray icons + SMAppService-equivalents.

Tray icon: WinUI 3 has no first-class system-tray API yet (as of late 2026). Use the standard `NotifyIcon` from `System.Windows.Forms` (still works in .NET 8 with `<UseWindowsForms>true</UseWindowsForms>` in the project file) for the icon + drop-down menu, and switch to WinUI 3 for the Settings + Wizard windows. This is a documented pattern.

## IPC client

`IpcClient.cs` uses `System.IO.Pipes.NamedPipeClientStream`. Connecting:

```csharp
var pipe = new NamedPipeClientStream(
    serverName: ".",                      // local machine
    pipeName: "mail-mcp-" + Environment.UserName,
    direction: PipeDirection.InOut,
    options: PipeOptions.Asynchronous);
await pipe.ConnectAsync(timeoutMs: 5000);
```

The IPC framing matches v0.1a: newline-delimited JSON-RPC 2.0. Models mirror Rust's snake_case via `[JsonPropertyName]` attributes:

```csharp
public sealed record AccountListItem(
    [property: JsonPropertyName("id")] string Id,
    [property: JsonPropertyName("label")] string Label,
    [property: JsonPropertyName("provider")] string Provider,
    [property: JsonPropertyName("email")] string Email,
    [property: JsonPropertyName("status")] AccountStatus Status);
```

The notification stream is exposed as `IAsyncEnumerable<DaemonNotification>` — natural for `await foreach` in WinUI event loops.

Failure modes:

- Pipe missing → `DaemonLauncher.Spawn()`. Retry connect with exponential backoff (capped).
- Stream EOF → reconnect; the notification stream emits a `Disconnected` element so view-models can show a banner.
- JSON-RPC error → typed `IpcError(int code, string message)` exception.

## Tray icon

```csharp
private NotifyIcon _icon;

void StartTray()
{
    _icon = new NotifyIcon
    {
        Icon = new Icon("Assets/tray.ico"),
        Visible = true,
        Text = "MailMCP",
    };
    _icon.ContextMenuStrip = BuildMenu();
    _icon.MouseClick += (_, e) => {
        if (e.Button == MouseButtons.Left) OpenSettings();

}
```

## First-run wizard

WinUI 3 `Window` with a `Frame` hosting six pages. Same six steps as macOS: Welcome → Add Account → OAuth → Permissions → Autostart → Configure AI client. State held in a `[ObservableProperty]`-decorated `WizardViewModel`; navigation via `Frame.Navigate`.

OAuth flow on Windows differs only in browser launch:
```csharp
Process.Start(new ProcessStartInfo { FileName = authUrl, UseShellExecute = true });
```

The autostart toggle uses the `Microsoft.Windows.AppLifecycle.AppInstance` API or, for plain background autostart, writes to `HKEY_CURRENT_USER\Software\Microsoft\Windows\CurrentVersion\Run` with the daemon's `.exe` path. (Registry path is simpler; documented as not requiring elevation.)

## Settings window

WinUI 3 `Window` hosting a `NavigationView`. Four pages mirror macOS:

- **Accounts** — list, add (reuses wizard's OAuth page as a dialog), remove
- **Permissions** — per-account ComboBox matrix (Read/Modify/Trash/Draft/Send × Allow/Confirm/Session/Draftify/Block)
- **General** — autostart toggle, Pause MCP toggle, MCP endpoint info card with Reveal/Copy, log file path with "Show in Explorer"
- **About** — version, uptime, redacted log tail, "Copy support bundle"

## Approval dialogs

`ApprovalCoordinator` subscribes to `approval.requested`. Each event presents a `ContentDialog`:

- Title: `"<Tool> on <account email>"`
- Body: same category-specific summary as macOS (Send: from/to/subject/body preview; Trash: count + sample subjects)
- Primary button: **Approve**
- Secondary button: **Reject**
- Auto-dismiss on `approval.resolved` from the daemon (admin CLI may resolve first)

ContentDialog requires a parent XamlRoot — for tray-only contexts where no window is open, the coordinator opens a hidden `Window` to host the dialog.

## Daemon spawning

`DaemonLauncher` runs at app launch:

1. Try `IpcClient.ConnectAsync()`. Connected → done.
2. Locate daemon at `Path.Combine(AppContext.BaseDirectory, "mail-mcp-daemon.exe")`. Spawn with `Process.Start` passing `--google-client-id` from `Environment.GetEnvironmentVariable("MAIL_MCP_GOOGLE_CLIENT_ID")` or a build-time-baked constant (see Build).
3. Wait up to 5s for `endpoint.json` to appear under `%LOCALAPPDATA%\mail-mcp\`.
4. Reconnect.

The daemon's PID lock (`fs2::FileExt::try_lock_exclusive`) works on Windows too (uses `LockFileEx`).

## Autostart

Two toggles, one for each:

- **Run MailMCP at login** — writes `HKCU\...\Run\MailMCP` to the tray .exe path.
- **Run daemon at login (without showing tray)** — writes `HKCU\...\Run\mail-mcp-daemon` to the daemon .exe path with `--google-client-id`. The tray launches on demand and connects.

For v0.1c we ship just the first toggle; the daemon-only autostart can come later.

## Build system

### Cargo ↔ MSBuild

A pre-build PowerShell script (`BuildScripts/build-daemon.ps1`) runs `cargo build --release --target x86_64-pc-windows-msvc -p mail-mcp-daemon` (and `aarch64-pc-windows-msvc` in CI matrix builds), then copies the `.exe` into the .NET output directory. Triggered via a `<Target Name="BeforeBuild">` MSBuild target in MailMCP.csproj.

For Debug, host-arch only; for Release, both architectures and a per-arch installer.

### Google OAuth client_id

Same approach as macOS: build-time constant from a `.local.props` file (gitignored) that overrides a default in `MailMCP.csproj`. CI populates from the `MAIL_MCP_GOOGLE_CLIENT_ID` secret.

## Distribution

### Code signing

EV or OV code-signing certificate from a CA (DigiCert, Sectigo). CI imports a base64-encoded `.pfx` from `WINDOWS_CERT_PFX_BASE64` and signs the `.exe` and `.msi` with `signtool sign /td sha256 /fd sha256 /tr http://timestamp.digicert.com /a`.

Until a cert is purchased, ship unsigned with a "right-click → Properties → Unblock" instruction (the spec already calls out unsigned-on-Windows as expected for v0.1).

### Installer

WiX 4 (`Product.wxs`) produces an MSI that:

- Installs to `%LOCALAPPDATA%\Programs\MailMCP\` (per-user, no admin elevation)
- Bundles `MailMCP.exe`, `mail-mcp-daemon.exe`, `mail-mcp-stdio.exe`
- Creates a Start Menu shortcut
- Adds an HKCU `Uninstall` entry

Per-user install matches the architecture: the daemon is per-user, runs at user scope, no service.

### Updates

Velopack (modern WinForms-friendly successor to Squirrel) for in-app updates. Or, as a simpler v0.1c starting point, just a "Check for Updates…" menu item that opens the GitHub Releases page in a browser. Velopack integration can come in v0.1c.1.

## Testing

- **Unit (`MailMCP.Tests`)** — `IpcClient` against a mock named-pipe server (a .NET task that runs `NamedPipeServerStream`); decoders for every IPC message type with golden JSON fixtures (shared with the macOS suite via `proto/ipc-schema.json`); view-models with a fake `IpcClient`.
- **UI (`MailMCP.UITests`)** — happy-path wizard test via WinAppDriver / Appium. Smoke-level only.
- **Integration** — extend `crates/mail-mcp-daemon/tests/integration_smoke.rs` with a Windows-runner-only path that pokes the named pipe directly.

CI: a new `Windows tray` job on `windows-latest` that runs `dotnet test`. UI tests run on the same job with WinAppDriver pre-installed.

## Phased plan

Three phases, mirroring v0.1b:

### Phase A — MVP (dogfood)

- VS solution + WinUI 3 project skeleton, build-daemon.ps1
- IpcClient + Codable models
- DaemonLauncher with bundled daemon spawning
- Tray icon + status menu
- First-run wizard (Welcome, Add Account, OAuth, Permissions defaults pass-through, Autostart, Configure AI client)
- Accounts pane (list, remove, add via wizard)
- Approval dialogs (ContentDialog)
- Unit tests for IPC client + view-models

### Phase B — settings completeness

- Permissions pane (full matrix)
- General pane (autostart toggle, pause MCP, endpoint info, logs)
- About pane
- "Run setup again" entry point
- WinAppDriver wizard smoke test

### Phase C — distribution

- WiX MSI installer
- Code signing in CI (gated on cert secrets, like macOS)
- (Optional) Velopack auto-update integration
- README install instructions

## Out of scope (future milestones)

- Linux tray (`v0.1d-linux-tray`)
- Microsoft Store / MSIX distribution
- M365 / IMAP providers (v0.2)
- Daemon-as-Windows-Service mode (would need elevation, breaks the per-user model)
