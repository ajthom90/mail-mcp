# mail-mcp v1.0 — Stable release readiness

## Goals

Ship the first stable release. Stable means:

- The IPC protocol, CLI, and on-disk file formats are versioned and forward-compatible.
- Public-API surface (the MCP tool set + Rust crate exports) is stable enough that breaking changes need a major version bump.
- All three tray apps + headless mode are dogfood-grade on their primary platforms.
- Telemetry / diagnostics / docs are good enough that a non-author can self-serve.

## Non-goals

- Mobile companion apps. (v1.x.)
- Web UI / browser extension. (Not a clear fit for this product.)
- Cloud account sync. (Local-first by design — see SECURITY.md.)
- Hosted SaaS. (No.)
- Localization beyond en-US. (v1.x — most strings live in `.po` files already, just no translations.)

## Required for v1.0

### Compatibility

- **IPC protocol versioning.** Each frame carries an implicit `protocol_version` derived from the daemon's `mail-mcp-daemon --version`. Tray apps refuse to connect to a daemon whose major version differs. Today the framing is implicit; v1.0 makes it explicit and adds a handshake `{"jsonrpc":"2.0","method":"hello","params":{"client_version":"..."}}` that the daemon answers with its version + capability flags.
- **On-disk format versioning.** `state.db` (sqlite), `accounts.toml`, `permissions.toml`, `saved-queries.toml`, `rules.toml` each carry a top-level `schema_version` field. Migration code lives in `crates/mail-mcp-core/src/migrations/`.
- **MCP tool stability.** The tool set as of v1.0 is frozen for the v1.x line. New tools may be added; existing tool signatures may not change.
- **Rust crate stability.** `mail-mcp-core` reaches semver-stable. The other workspace crates remain internal.

### Quality

- **Test matrix expansion.** Today: cross-compile checks + unit tests + Windows-tray dotnet test + linux-tray meson test + macOS xcodebuild. v1.0 adds: Linux runner with libsecret-keychain test (currently `keychain-tests` feature is local-only); Windows runner with credential-manager test; CalDAV / IMAP integration smoke; and OAuth-refresh-flow tests with provider mocks.
- **Soak test.** A dedicated weekly CI job runs the daemon + tray for 24 hours against mock providers, verifies no leaks (RSS bound), no crashes, no log spam.
- **Crash reporting.** Opt-in (default off) via the existing tracing pipeline → minidumps → user-controlled local path. No third-party crash service.
- **Code coverage.** Goal is 70% line coverage on `mail-mcp-core`. Tracked in CI but non-blocking on PRs.

### Security

- **Threat model document.** Update `SECURITY.md` to reference an explicit threat-model doc (`docs/threat-model.md`) covering: local same-user attacker (out of scope per design), local cross-user attacker (mitigated via 0600 socket + per-user pipe), token theft (mitigated via OS keychain), MCP host compromise (mitigated via per-action approvals), provider-side abuse (mitigated via OAuth scope minimization).
- **Dependency audit.** `cargo audit` clean on every CI run. Known-issue exceptions documented in-repo.
- **Reproducible builds.** `cargo build --release` with `--locked` should produce byte-identical binaries given the same source. Verify in CI.
- **Code signing.** macOS Developer ID + notarization. Windows EV certificate. Linux: Flathub signing.

### Distribution

- **macOS .dmg** signed + notarized via the existing GitHub Actions release workflow. Sparkle update channel published at appcast.xml.
- **Windows .msi** built via WiX 4 + signed with the EV cert. Auto-update via WinSparkle (NOT WindowsAppSDK auto-update — too tied to MSIX).
- **Linux .tar.gz** + Flatpak. Flathub submission pre-flighted.
- **Headless / Docker.** Pre-built Docker image of the daemon for self-hosted / NAS users. Tag policy: `:1`, `:1.2`, `:1.2.3`, `:latest`.
- **Homebrew formula** for the headless daemon (Mac) — `brew install mail-mcp-daemon`.

### Documentation

- **Quickstart.** "From `brew install` to first MCP tool call in 5 minutes" — separate doc per platform.
- **Power-user reference.** Saved-query expression syntax. Rule engine reference. Permission policy semantics.
- **Provider matrix.** What works / doesn't work per provider (Gmail, M365, IMAP).
- **MCP host config recipes.** Claude Desktop, Claude Code, Cline, custom hosts.
- **Troubleshooting.** Common errors + resolutions. Log file locations. How to file a bug report.
- **Architecture overview.** A doc that takes someone from "what is this" to "I could contribute" in an hour. Heavy diagrams.

### Operational

- **Telemetry (opt-in).** Anonymous usage counters via the existing tracing pipeline → optional aggregation server. Counts: tool invocations by name, error rates, provider mix, OS / version mix. NO message content, NO email addresses, NO subject lines.
- **In-app feedback.** "Send feedback" menu item in each tray, opens a pre-filled GitHub issue.
- **Release cadence.** Patch releases on demand. Minor releases monthly. Major releases on real breaking changes (rare post-1.0).

## Pre-1.0 cleanup

Things known-to-be-ugly that should land before v1.0:

- Tray-app workspace consolidation. Currently `tray-app/`, `tray-app-win/`, `tray-app-linux/` are independent build trees with duplicated build-daemon scripts. v1.0 unifies via a single shared bash/PowerShell script that the platform builds invoke.
- IPC schema generator. The IPC types are mirrored by hand in Rust / Swift / C# / C. v1.0 introduces `proto/ipc-schema.json` + per-language code generators (`xtask gen-ipc`) so the mirrors stay in sync.
- Permission migration. The default-policy table changed in v0.2 (added `confirm` for `modify`); migrations should normalize old `permissions.toml` to the v1.0 shape.
- Logging cleanup. Today the daemon logs at `info` by default; v1.0 trims that to `warn` for production binaries with `RUST_LOG=mail_mcp=info` for verbose.
- Error message audit. User-facing strings should consistently say what failed + what to try. Right now they're a mix of technical + actionable.

## Out of scope (post-v1.0)

- v1.1: Mobile companion apps (iOS / Android). Read-only initially; full feature parity is a long-term project.
- v1.2: Web UI for power-user rule editing.
- v1.3: Multi-host MCP fan-out (one daemon serves multiple Claude Desktops on the same machine — useful for shared family / household setups).
- v2.0: Re-evaluation of the MCP-tool boundary as MCP itself evolves.
